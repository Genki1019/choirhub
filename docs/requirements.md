## 1. プロジェクト概要

### 1.1 プロダクト名

#### ChoirHub

### 1.2 概要

合唱団の運営に関わるすべての業務をひとつのWebサービスで完結させる、マルチテナント型の合唱団向け運営支援 SaaS。

### 1.3 解決する課題

現在の合唱団運営は複数のツールが乱立しており、情報が分散している。

| 課題 | 現状 | ChoirHub での解決 |
|------|------|-----------------|
| スケジュール・出欠管理 | 伝助など外部ツール依存 | 統合出欠カレンダー |
| 楽譜の配布・管理 | メール添付・紙配布 | クラウド楽譜管理 |
| 本番出演調査 | Googleフォーム等 | オンステ調査フロー |
| 連絡・アーカイブ | メーリングリスト | 送受信履歴の一元管理 |
| チケット販売管理 | Excelスプレッドシート | チケット配布・集計 |

---

## 2. 目的・背景

### 2.1 ビジョン

「合唱団の運営担当者が、ChoirHub だけで団の業務が完結できる状態をつくる」

### 2.2 初期ターゲット

男声合唱団。ただしパート構成をカスタマイズ可能にすることで、混声・女声・学生合唱にも展開できる設計とする。

## 3. ターゲットユーザー

### 3.1 主要ペルソナ

#### ペルソナ A: 運営担当者（技術系・楽譜がかり）

- 年齢: 20〜40代
- 役割: スケジュール管理・楽譜購入配布・連絡業務を担う
- 課題: 複数ツールの管理が煩雑。情報がメールとスプレッドシートに散在している

#### ペルソナ B: 一般団員

- 年齢: 20〜60代（幅広い）
- 役割: 練習参加・出欠回答・楽譜の確認
- 課題: 「次の練習いつ？誰が来る？」をすぐ確認したい。楽譜をスマホで見たい

#### ペルソナ C: 客演・体験参加者

- 役割: 一時的な参加者
- 課題: 団員と同じツールを使えるが、機密情報へのアクセスは不要

### 3.2 複数団体所属

1人のユーザーが複数の合唱団に所属できる。

- 例: 男声合唱団 A と混声合唱団 B の両方に所属する団員
- ログイン後に所属団体が複数ある場合は、**団体選択画面**（`/select-org`）を経由して入団先を選ぶ
- 一度選択した後もサイドバーの**団体切替メニュー**からいつでも切り替えられる
- 各団体での**ロール・パート・プロフィールは独立**して管理する（「団体Aでは tech、団体Bでは member」が可能）

> **認証モデルの分離**: ユーザーアカウント（`User`）と団体所属（`Member`）は別エンティティとして管理する。認証はメールアドレスで行い、その後どの団体に入るかを解決する。

## 4. ロール・権限定義

### 4.1 ロール一覧

| ロール | 英名 | 説明 | 想定人数 |
|--------|------|------|---------|
| 最高管理者 | `admin` | 全権限。メンバー管理・設定変更 | 1〜2名 |
| 技術系 | `tech` | 選曲・ステージ構成・スケジュール管理 | 複数名 |
| 指揮者 | `conductor` | `tech` と同等の権限（将来的に分離可能な設計とする） | 1〜2名 |
| 楽譜がかり | `score` | 楽譜購入・価格設定・配布・閲覧許可管理 | 複数名 |
| チケット担当 | `ticket` | チケット配布設定・全体集計・集金確認・パートレース公開管理 | 1〜複数名 |
| 会計係 | `finance` | 支出管理（会場費・報酬）・団員支払い記録 | 1〜2名 |
| 一般 | `member` | 閲覧・出欠回答・自分のチケット販売入力 | 多数 |
| 客演・共演者 | `guest` | スケジュール・楽譜閲覧・出欠回答 | 少数 |
| 体験 | `visitor` | 共有アカウント。全楽譜PDFをブラウザで閲覧可（MIDI・音声不可）。パート未設定。各種一覧（メンバー・出欠・購入者・チケット等）には表示されない | 1アカウント |

> **注**: `guest` および `visitor` は、メンバー一覧・出欠表・楽譜購入者一覧・オンステ調査・チケット配分一覧などの「一覧表示」に自分のアカウントが表示されない。他の団員のプロフィールや情報は閲覧可。

#### 補足

- `tech` と `score` は兼任可能（複数ロール付与）
- `conductor` は現時点で `tech` と同等。将来的に指揮者固有の権限（テンポ指定・演奏指示管理など）が生じた際に分離する
- `ticket` / `finance` / `conductor` は `member` と兼任して使用する（例: `["ticket", "member"]`）

### 5.2 ホーム

**目的**: ログイン直後に「今の団の状況」を一目で把握できる

#### 表示要件

- 直近の練習・本番予定（次回イベント3件程度）
- 自分の未回答出欠のバッジ表示
- 今月の飲み会幹事パート（チケットレース結果に基づく）※ Phase 2 以降で実装予定（MVP では非表示）
- 直近のメール（最新1〜2件のタイトル）

### 5.3 メンバー管理

**目的**: 団員情報の一元管理と自己プロフィール編集

#### 機能要件

- メンバー一覧（カード表示・リスト表示の切り替え）
- パート・ステータス・ロールによるフィルタリング
- 顔写真付きプロフィールカード表示
- 招待メールによる入団フロー（URLトークン方式）
- 自分のプロフィール編集（bio・job・interests・顔写真）

#### メンバーステータス

| ステータス | 説明 |
|-----------|------|
| `active` | 現役団員 |
| `offstage` | 休団中 |
| `alumni` | OB/OG |
| `suspended` | 一時停止 |

### 5.4 スケジュール・出欠管理

**目的**: 伝助ライクな出欠カレンダーで練習・本番の参加状況を管理する

#### 機能要件

- 月カレンダー表示（一覧）
- イベント詳細・出欠表（伝助ビュー）
  - 縦軸: メンバー（パート順）
  - 各セルで○△✕を選択
  - ページ全体備考（管理者入力）
  - セルごとの個人メモ（到着・退席予定時刻など）
  - パート別集計行（○N / △N / ✕N）
- 回答締切設定・締切後ロック
- イベント種別: 練習（`rehearsal`）/ 本番（`concert`）/ 会議（`meeting`）/ その他（`other`）

#### 出欠ステータス

| 記号 | ステータス | 意味 |
|------|-----------|------|
| ○ | `attending` | 参加 |
| △ | `maybe` | 遅刻・早退・未定 |
| ✕ | `absent` | 欠席 |
| — | `undecided` | 未回答 |

### 5.5 楽譜・MIDI管理

**目的**: 楽譜ファイルの安全な保管・配布と価格管理

#### 機能要件

- 楽譜一覧（演奏会別グループ表示・演奏会未定セクション）
- ファイルアップロード（PDF・MIDI・その他）
- アクセスレベル制御（委嘱初演曲はオンステ確定者のみ閲覧可）
- ファイルダウンロードはサーバー経由でストリーミング配信（R2への直リンク禁止。日本語ファイル名は RFC 5987 エンコード）
- **団内配布価格の設定・閲覧**:楽譜がかり（`score+`）が登録、`member` 以上が金額を参照可
- 購入記録の管理（楽譜がかりのみ）
- 演奏会との紐付け（演目を通じて紐付く）

#### 楽譜がかりの業務フロー

```text
楽譜がかりが楽譜を一括購入
  → 購入価格・頒布価格を登録
  → 各団員の購入記録を登録（購入した団員を記録）
  → 購入記録のある団員のみ楽譜PDFのDLが許可される
```

> **設計意図**: 楽譜の閲覧・DL権限は「ロール」と「購入記録の有無」で制御する。
>
> - **体験（visitor）**: 団体が管理する共有アカウント。全楽譜（access_level 問わず）のPDFをブラウザ標準ビューワーで閲覧可。MIDI・音声は不可。ダウンロード専用ボタンは非表示（ブラウザの保存機能は利用可）。
> - **一般団員（member/guest）**: access_level（public/restricted/secret）に関わらず、楽譜がかりが登録した購入記録がある楽譜のみDL可。secret楽譜は購入記録があっても閲覧不可。
> - **特権ユーザー（admin/score/tech/conductor）**: access_level・購入記録に関係なく全ファイルにアクセス可。

#### アクセスレベル

| レベル | 説明 |
|--------|------|
| `secret` | 管理者・特権ユーザーのみ閲覧可（体験はPDF閲覧可）。一般団員は購入記録があっても不可 |
| `restricted` | 購入記録のある団員はDL可。体験はPDF閲覧可 |
| `public` | 購入記録のある団員はDL可。体験はPDF閲覧可（publicでも購入記録必須に変更）|

### 5.6 本番・オンステ管理

**目的**: 演奏会の構成管理と出演調査・確定・立ち位置（フォーメーション）管理フロー

#### 階層構造

```text
演奏会（Concert）
  └─ ステージ（Stage: 第1ステージ、第2ステージ…）
       ├─ 演目（Program: 曲名・曲順）
       │    └─ 楽譜（Score）への紐付け
       └─ フォーメーションパターン（FormationPattern）
            ├─ 指揮・ピアノ・カスタム枠（FormationBox）
            └─ 山台・枠への配置（FormationSlot）
```

#### 機能要件

演奏会・ステージ・演目の管理（**`admin` のみ**）:

- 演奏会の新規登録（タイトル・開催日・会場）
- ステージの追加・名前のインライン編集・↑↓ボタンによる並び替え
- 演目の追加（2モード）:
  - **新規作成モード**: タイトル・作曲者・編曲者を入力して楽譜を新規登録しつつ追加
  - **既存から選ぶモード**: 楽譜一覧から検索して選択し追加
- 演目の↑↓ボタンによる並び替え
- **演目の移動・コピー**: ステージ内の演目を別演奏会のステージへ移動またはコピー。「演奏会未定」へ移動（プログラムから削除、楽譜は残る）も可能

オンステ調査フロー（**`tech` 以上**が調査の開設・締切・確定操作を行う）:

1. tech 以上が調査を開設（演奏会に対して複数回・複数調査を作成可能。例: 一次調査・二次調査）
2. 各メンバーが**ステージ単位**で出席/欠席/未回答の3択で回答（演目単位ではない。スケジュール出欠の `maybe`（△・遅刻早退未定）とは異なり、オンステ調査に △ の選択肢はない）
3. 調査を締め切ると、回答内容が自動的にオンステ確定（`OnStageAssignment`）へ反映され、演奏会ステータスが `confirmed` になる
4. 複数調査がある場合、締切後でも tech 以上は任意の調査を選んで明示的に反映し直せる（フォーメーション編集画面から「フォーメーションに反映」）
5. オンステ確定後はメンバーに公開（閲覧のみ、`off` になったメンバーは配置からも自動的に外れる）

フォーメーション（立ち位置）管理（**`tech` 以上**が編集、閲覧はオンステ確定済みの全メンバー）:

- ステージごとに複数の「パターン」を作成・名称変更・削除・並び替え可能（曲によって立ち位置を変える場合などに使う）
- パターン作成時、指揮・ピアノの枠が自動的に用意される。ソロ・楽器などの任意の「カスタム枠」も追加可能
- 山台の段数を増減、段ごとに人数を配置（列数はオンステ人数から自動算出）、段を半人分ずつ互い違いにずらす「千鳥配置」の切り替え
- ピアノの表示位置（中央 / 下手）を切り替え可能
- 出演確定メンバー・客演（団員外の出演者を名前のみで追加）を、山台の任意のマスまたは指揮・ピアノ・カスタム枠へドラッグ&ドロップで配置。丸バッジ内の表示名は個別に上書き可能
- 編集画面とは別に、確定した配置を見やすく表示するプレビュー画面を切り替え表示できる

### 5.7 メーリス管理

**目的**: 団内メールの送信と履歴の永続アーカイブ

#### 機能要件

- 宛先絞り込み（パート別・ロール別・全員）
- テンプレートの保存・呼び出し
- 送信履歴一覧（件名・送信日時・送信者・宛先数）
- 本文全文の閲覧（検索可能）
- 外部メーラーで削除されても ChoirHub 上に残る

### 5.8 チケット販売管理

**目的**: 演奏会チケットの配布・回収・集計と、パート間の販売レース

#### 機能要件

- 席種（一般・学生・招待など）ごとのチケットバッチ管理
- 団員別の配布枚数設定（ticket担当 / admin のみ）
- **自分のチケット販売入力**（全団員が自分の分のみ入力可）
  - 販売枚数（一般・学生・その他）
  - 返却枚数
  - 集金済みかどうか（※集金担当は要検討）
- 他メンバーの状況は **非表示**（自分の入力画面のみ見える）
- **チケット担当 / admin 向け集計画面**
  - 全員分の配布・販売・返却・集金状況を一覧表示
  - パート別累計販売枚数・販売率の集計
- **パートセールスレース**
  - ticket担当 / admin のみ閲覧・管理
  - 「全体公開」ボタンを押すまで一般団員には非表示（ネタバレ防止）
  - 公開後は全団員がパート別ランキングを閲覧可能
  - 飲み会幹事パート発表イベントとして活用

### 5.9 会計・費用管理

**目的**: 団の収支（支出・徴収）を一元管理し、会計係の記録作業を最小化する

---

#### 5.9.1 徴収方式の設定（admin のみ）

団ごとに以下の2方式から選択できる。設定は `/[org]/settings` で変更可能。

| 方式 | 説明 | Collection の自動生成タイミング |
|------|------|-------------------------------|
| `per_rehearsal`（練習ごと） | 練習に参加した団員から毎回徴収 | 練習イベント作成時に自動生成 |
| `monthly`（月会費制） | 毎月固定額を全員から徴収 | 会計係が「今月の会費を発行」を実行 |

- `defaultFeeAmount`（デフォルト徴収金額）も設定可能。Collection 自動生成時の初期値として使用される

---

#### 5.9.2 徴収管理（finance / admin のみ）

#### Collection（徴収）の自動生成

- `per_rehearsal` モード: 練習イベント作成と同時に Collection が自動生成される
- `monthly` モード: 会計係が「今月の会費を発行」ボタンを実行すると生成される
- 生成直後に **全アクティブ団員**（`status = active` かつ `visitor` ロール除く）の `CollectionPayment`（`status = pending`）が自動作成される

**Collection は手動でも作成可能**（合宿費・飲み会代・本番参加費など）

#### 支払い記録

- 会計係が団員一覧を見ながら、支払いが来た団員の `CollectionPayment.status` を `paid` に更新
  - 実際の支払額・支払方法（現金 / PayPay / 振込）・支払日を記録
- 欠席など徴収不要な団員は `waived`（免除）に変更
- 未払い一覧を一覧で確認できる

#### 閲覧権限

- finance / admin: 全徴収の作成・編集・支払い記録
- member 以上: 自分の `CollectionPayment` 状況のみ閲覧可

---

#### 5.9.3 支出管理（finance / admin のみ）

#### 機能要件

- 支出の登録・編集・削除
  - タイトル・カテゴリ（ExpenseCategory から選択）・金額・支払方法・支払日・メモ
- 練習・本番イベントへの紐付け（任意）
- 支出一覧（月次フィルタ・カテゴリフィルタ）

#### 支出カテゴリ（団ごとにカスタマイズ可、デフォルト値）

| カテゴリ名 | 例 |
|-----------|---|
| 会場費 | 練習室レンタル代 |
| 指導者謝礼 | 指揮者・ボイトレ・ピアニスト報酬 |
| 楽譜費 | 楽譜一括購入代 |
| 合宿費 | 宿泊・食事費 |
| 本番費 | ホール・印刷・衣装等 |
| 飲み会費 | 打ち上げ会場代等 |
| 雑費 | 消耗品・備品等 |

カテゴリは設定画面から追加・編集・削除可能。

---

#### 5.9.4 収支サマリー（finance / admin のみ）

#### 表示内容

- 月次フィルタで期間を選択
- 収入合計（徴収済 CollectionPayment.amount 合計）
- 支出合計（Expense.amount 合計）
- 収支差額
- 徴収一覧・支出一覧（その月のものを一覧表示）

**収入の内訳**（会計集計に含めるもの）

- CollectionPayment（`status = paid`）の合計
- ScorePurchase.amount の合計（楽譜代徴収）
- TicketAllocation（`isCollected = true`）の合計（チケット代回収）はチケット機能側で管理し、収支サマリーには参考表示

### 5.10 設定

**目的**: 団ごとの情報・パート構成のカスタマイズ

#### 機能要件

- 団体情報の編集（名前・ロゴ等）
- パート管理（追加・編集・並び替え）

## 6. 非機能要件

### 6.1 セキュリティ

| 要件 | 内容 |
|------|------|
| マルチテナント分離 | 全DBクエリに `orgId` を付与し、テナント間のデータ参照を防ぐ |
| ファイルアクセス制御 | S3/R2への直リンク禁止。ダウンロードは Presigned URL を発行 |
| 認証 | セッションベース認証（Lucia v3）|
| 権限チェック | APIミドルウェア層で実施。各エンドポイントの重複チェックは不要 |

### 6.2 ユーザビリティ

| 要件 | 内容 |
|------|------|
| レスポンシブ対応 | スマートフォンでの出欠回答・楽譜閲覧が快適にできること |
| 操作の直感性 | 合唱団員（IT非専門家）が説明なしで基本操作できること |

### 6.3 可用性・パフォーマンス

| 要件 | 内容 |
|------|------|
| 目標稼働率 | 99%以上（個人開発のため SLA は設けない）|
| 出欠表の応答速度 | 50名規模の出欠表が 2秒以内に表示されること |

### 6.4 拡張性

- パート構成をカスタマイズ可能にし、男声以外の合唱形態にも対応できること
- 将来的な Stripe 決済・Google Calendar 連携を考慮したデータ設計とすること

## 8. 用語定義

| 用語 | 定義 |
|------|------|
| テナント / Organization | ChoirHub を利用する合唱団の単位 |
| オンステ | 本番（演奏会）のステージに出演すること |
| オンステ調査 | 演奏会の各演目に対して出演可否を回答するアンケート |
| 伝助ビュー | 縦軸: メンバー / 横軸: イベントで出欠を管理する表形式UI（伝助 <https://densuke.biz/> を踏襲）|
| パートレース | 演奏会チケットの販売枚数をパートごとに競うランキング機能。公開前は ticket / admin のみ閲覧可 |
| Presigned URL | S3/R2 のファイルに期限付きで直接アクセスできる署名付きURL |
| 委嘱初演曲 | 団が作曲家に依頼して初めて演奏する曲。楽譜の取り扱いに制限を設ける |
| 場所代 | 練習会場を使用するために団員が払う参加費 |
| 支出 | 会場費・指揮者報酬・ボイストレーナー報酬・ゲスト報酬など、団の運営にかかる費用 |
